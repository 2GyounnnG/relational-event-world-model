## Decision

The next project entry should be treated as **provisional**:

**Step32 fixed-protocol mechanism-gap planning is the leading next-phase candidate, but it should be formalized only after a dedicated Step32 source-of-truth consolidation memo.**

## 5. Recommended Next-Phase Direction

The recommended next-phase direction is:

**Step32 source-of-truth consolidation, followed by a fixed-protocol mechanism-gap decision if the consolidation confirms the current Step32 status.**

This is one direction, not multiple branches. It does not continue Step33. It does not continue Step22-29. It does not continue Step30 or Step31 micro-lines. It does not start backend transfer. It does not change defaults.

Why this is evidence-bounded:

- README already names Step32 as the active next-phase question.
- Step30 docs say the next move after Step30 should be a stronger synthetic observation/representation substrate, not more Step30 micro-tweaks.
- Step31 docs validate synthetic multi-view as a stronger bridge family and retain `step31_simple_late_fusion`.
- Step31 consolidation recommends moving to Step32 as a synthetic rendered/image-like bridge probe.
- Step32 status docs show Step32 has meaningful isolated candidate references.
- Step32 status docs also explicitly state the line is not formally promoted, remains threshold-sensitive, remains below Step30 rev6, and keeps backend transfer closed.

Therefore Step32 is the leading candidate, but the next memo should consolidate Step32 status before it is treated as the formal project entry point after Step33.

## 6. Why This Is Better Than Reopening Local Stalled Lines

Reopening Step33 would be low value because the line has already isolated its plateau. The benchmark is healthy and proposal-side signal is viable, but the current implementation family is not candidate-ready. The final Step33 check shows the trained combined row is stable, yet it does not beat the composed diagnostic assembly or close the stratified-test gap to `spring_neighbor_scope`.

Reopening Step22-29 would also be low value. That line ended cleanly with a retained branch candidate: `Step26 proposal + RFT1`. The branch has real strongly-interacting changed/add value, but it is not the broad noisy default because it gives back too much full/context stability. More local joint, anchored, proposal-only, or threshold tweaks would revisit an already-mapped tradeoff.

Reopening Step30 micro-lines would be low value because Step30 consolidation says later variants did not clearly beat rev6 and mostly repeated a recall-vs-precision tradeoff.

Reopening Step31 learned-fusion micro-lines would be low value because Step31 consolidation keeps `step31_simple_late_fusion` as retained and parks Step31c-e.

Starting from consolidated defaults is cleaner. The project already has stable broad defaults, targeted references, and a retained noisy interaction-aware branch candidate. The next entry point should ask a consolidated observation/representation question under those defaults rather than continue a saturated local line.
