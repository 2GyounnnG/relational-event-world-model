# Step32 Synthetic Rendered / Image-Like Bridge Probe

Step31 learned-vs-late fusion is now consolidated: `step31_simple_late_fusion`
remains the retained backend-transfer reference, and
`step31c_agreement_damped_encoder` remains the retained learned-fusion
diagnostic reference. The Step31c-e learned-fusion micro-line is parked.
Adapter/interface work remains premature.

## Step32 Question

Step32 tests whether a more perceptual but still fully synthetic observation
family can support structured recovery. The probe remains recovery-first and
does not use real-world images, real-world labels, hypergraphs, LLM integration,
backend joint training, or backend redesign.

## Observation Family

The new rendered observation is generated by
`data/generate_step32_rendered_observation_data.py`.

Each sample keeps the original clean graph/event labels, then creates one
synthetic rendered observation family:

- two small 32x32 rendered views by default
- node marks drawn from corrupted weak slot observations
- weak relation lines drawn from corrupted relation hints
- support/evidence lines drawn from corrupted pair-support and evidence hints
- witness/visibility channel drawn from corrupted signed pair witness plus
  occlusion marks
- view-dependent layout jitter, node dropout, line dropout, false lines, pixel
  noise, and partial rectangular occlusion

The clean adjacency is not rendered directly. It is retained only as the
structured recovery target and evaluation label.

## Model

The new bridge model is `models/encoder_step32.py`.

It uses a small shared CNN over rendered images, samples node descriptors at the
known synthetic layout coordinates, samples line descriptors between node pairs,
then decodes node state/type and pairwise edges with lightweight MLP heads.

This is intentionally small and inspectable. It is not a large vision stack.

## Baselines

The evaluation script `train/eval_step32_rendered_bridge.py` compares:

- Step30 rev6 reference constants
- Step31 retained `step31_simple_late_fusion` reference constants
- `trivial_rendered_relation`, a direct line-intensity decoder
- `rendered_projection_baseline`, a hand-weighted relation/support decoder
- `step32_rendered_bridge`, the learned tiny CNN bridge

It reports overall, clean, noisy, and event-family edge metrics, plus rendered
missed-edge/rescue diagnostics.

## Backend Policy

Backend transfer is not automatic. The Step32 evaluation gate recommends a
focused backend transfer only if rendered recovery clearly beats Step30 rev6 and
learned rendered decoding is materially above trivial rendered decoding.

## Run Commands

First generate base graph-event splits if they are absent. Then generate Step32
rendered splits, train, and evaluate:

```powershell
python data/generate_graph_event_data.py
python data/generate_step32_rendered_observation_data.py --generate_default_splits
python train/train_step32_rendered_bridge.py --train_path data/graph_event_step32_rendered_train.pkl --val_path data/graph_event_step32_rendered_val.pkl --save_dir checkpoints/step32_rendered_bridge
python train/eval_step32_rendered_bridge.py --data_path data/graph_event_step32_rendered_test.pkl --step32_checkpoint checkpoints/step32_rendered_bridge/best.pt --output_dir artifacts/step32_rendered_bridge_probe
```

In this workspace snapshot, Python/PyTorch and the prior generated
data/checkpoints are not present, so the recovery table is expected to be filled
by the first runnable local execution.
